2 · Diagnosis

Refuted

Chat-template drift (the writeup's lead hypothesis)

  • sft_data.py:44 returns a bare messages column; TRL applies Qwen's own chat template at train time (sft.py:128-134, no formatting_func). Train-time rendering == eval-time rendering — both call apply_chat_template. Nothing to drift.
  • Tokenizer (verified): eos=<|im_end|> (151645), pad=<|endoftext|> (151643), pad ≠ eos. The eos-suppression footgun does not apply; the base.py:110 pad is None fallback never fires.
  • No hand-rolled User:/Assistant: concatenation exists anywhere in the pipeline.
Primary cause

No assistant-only loss mask

Verified against installed trl 1.4.0: SFTConfig.assistant_only_loss defaults to False; sft_qwen05b.yaml:20-37 never sets it; sft.py:128-134 attaches no DataCollatorForCompletionOnlyLM.

→ Loss is computed over the entire rendered conversation, including every user and system turn. UltraChat-200k is heavily multi-turn, so a large share of the gradient teaches the 0.5B model to generate user questions — capacity spent on the wrong objective, degrading general behavior mildly and uniformly. DPO would inherit this bug, so it must be fixed regardless.

Secondary (possibly dominant)

Expected forgetting

The base is Qwen2.5-0.5B-Instruct, already aligned by Qwen. One epoch of LoRA over 5k general-chat rows on an already-tuned model routinely drops benchmarks a couple points with no bug present. The observed magnitude sits squarely in this regime.

Honest implication: fixing the mask is correct and necessary, but it may move SFT to "≈ base," not necessarily "> base." The Phase 1 success criterion (sft > base on IFEval) may be optimistic for SFT-on-Instruct independent of this bug — worth a separate decision, not a hyperparameter band-aid.
